## 系統概覽

```
all_cards.json          （來源資料，363 筆）
       │
       ▼
build_data.py           （本地執行，不進版本控制 — .gitignore: *.py）
  ├── translate_skill()     26 段有序規則 (section 1 → section Z)
  ├── 後處理 fixup          語序修正、殘留清理
  └── MANUAL_OVERRIDES      最終覆蓋層（個別卡片）
       │
       ▼
cards_data.js           （自動產生，進版本控制）
       │
       ▼
index.html 載入 cards_data.js → 使用者瀏覽器顯示
```

- **`build_data.py`**：本地執行，因 `.gitignore: *.py` 不進版本控制。  
  維護翻譯規則後需手動執行並將產生的 `cards_data.js` commit。
- **`cards_data.js`**：268 KB，364 張卡片，**請勿手動編輯**。
- **唯一鍵**：`image_file`（如 `HV-P01-010-S.webp`），364 張全唯一。  
  `card_no` 有重複（僅 296 個唯一值），不能用作唯一鍵。

## 翻譯管線架構

`translate_skill()` 函式依序執行以下 26 段規則。  
**規則順序至關重要**：長串/特殊模式必須寫在短串/一般模式之前。

| 段落 | 說明 | 關鍵規則範例 |
|------|------|--------------|
| 1 | 特殊括號標籤（帶數字） | `[=フェイント(N)]` → `[=佯攻(N)]` |
| 2 | 觸發標籤 | `[=アタック]` → `[=攻擊]` |
| 2.5 | 完整短語保護 | `以下から(N)つを選んで使える` → `選擇以下其中N項使用`（必須在 `で使える` 之前！） |
| 3 | 複合詞組（含手牌棄置系列） | `手札からXをドロップすれば使える` → `從手牌棄置X後可發動` |
| 3b | 指示詞 | `その後` → `之後`（必須在 `その` → `該` 之前） |
| 4 | 區域複合詞 | `ブロックエリア` → `攔網區` |
| 5 | 單詞替換 | `手札` → `手牌`、`スキル` → `技能` |
| 6 | 代名詞 | `自分の` → `自己的`、`相手の` → `對手的` |
| 7 | 費用/發動 | `(N)Guts払えば使える` → `支付N Guts後可發動` |
| 8 | 棄牌 | `ドロップすれば使える` → `棄置後可發動` |
| 9 | 抽牌 | `(N)枚を引く` → `抽N張牌` |
| 10 | 牌庫操作 | `デッキに戻してシャッフル` → `放回牌庫並洗牌` |
| 11 | 加入手牌 | `手牌に加えるたび` → `加入手牌時，` |
| 12 | 數值變化 | `に+(N)` → ` +N`；`さらに值 +N` → `再+N值` |
| 13 | 出場 | `登場させられない`（優先）→ `不能出場`；`が登場するたび` → `出場時，` |
| 14 | 條件判定 | `(N)以上の場合` → `N或以上時，` |
| 15 | 各種動詞 | `してもよい` → `（可選擇）`；`ロスト` → `失分` |
| 16 | 數量詞 | `(N)枚まで` → `最多N張` |
| O | 區域 + から | `XXエリアから` → `從XX區` |
| P | 區域 + に | `XXエリアに` → `在XX區中` |
| Q | 值比較（=） | 注意：`値が(N)或以上/以下` 必須在 `値が(N)` → `値=N` 之前 |
| R | 人物名稱 | 球隊/人名替換 |
| S | 後處理 fixup | 語序修正、殘留清理（詳見下方問題列表） |
| Y | 標點符號整理 | 清除多餘助詞、統一逗號 |
| Z | 防禦值 | `ディフェンスポイント` → `防禦值` |

## 重要設計原則

1. **特殊/長串優先**：新增規則時，特殊或較長的模式必須寫在一般或較短的模式之前，否則短串會先把長串「吃掉」一部分，導致後面規則無法匹配。

2. **後處理 fixup（section S）**：用來修正「語序翻轉」類問題。當規則本身無法避免語序錯誤（因為 regex 逐步套用的性質），改在最後用後處理統一修正。

3. **MANUAL_OVERRIDES 最後套用**：在 `translate_skill()` 執行完畢後才套用覆蓋層，確保個別卡片的手工翻譯不被規則破壞。

4. **`たび` 觸發詞原則**：`たび`（每次觸發）一律翻成「當…時，」，不加「每當/每次」。中文卡牌文本中「當X出場時，Y」已隱含每次觸發的語意。

5. **`image_file` 作為 MANUAL_OVERRIDES 的 key**：因為 `card_no` 不唯一（有 S/H 版本等），必須用 `image_file` 作為識別鍵。

## MANUAL_OVERRIDES 使用說明

```python
MANUAL_OVERRIDES = {
    'HV-P01-010-S.webp': '翻譯結果字串',
    'HV-P01-011.webp':   '翻譯結果字串',
    # ...約 130 筆
}
```

- **使用時機**：`translate_skill()` 規則無法修正的個別卡片（結構太特殊或組合太複雜）。
- **套用時機**：在 `translate_skill()` 完整執行之後才套用，**完整覆蓋**自動翻譯結果。
- **保留原則**：若後來新增規則已能正確翻譯某卡片，MANUAL_OVERRIDES 的對應條目仍可保留，不影響正確性（會覆蓋自動翻譯結果，效果相同）。
- **目前規模**：約 130 筆，集中在語序複雜、雙觸發、特殊條件的卡片。

## 已修正的翻譯問題（共 15 項）

> 以下問題均已在 `build_data.py` 中修正。記錄於此供未來維護時參考，避免重複踩坑。

---

### 問題一｜規則順序衝突：`その後` vs `その`

- **症狀**：「その後」翻成「該後」而非「之後」
- **原因**：`その → 該` 規則先執行，把「その後」中的「その」吃掉
- **解法**：`その後 → 之後` 必須寫在 `その → 該` 之前（特殊長串 > 一般短串）
- **位置**：`build_data.py` section 3b（`その後` 排在 `その` 前一行）

---

### 問題二｜規則順序衝突：`登場させられない` vs `登場させ`

- **症狀**：「登場させられない」翻成「使其出場至，不能」（拆壞）
- **原因**：`登場させ → 使其出場` 先把前半吃掉，`られない` 孤立殘留
- **解法**：`登場させられない → 不能出場` 必須寫在所有 `登場させ` 規則之前
- **位置**：`build_data.py` section 13

---

### 問題三｜觸發詞 `たび` 不應翻成「每當/每次」

- **症狀**：「が登場するたび」→「每次出場時」（語氣繁冗）
- **原因**：`たび` 直譯為「每次」，但中文卡牌文法中「當X出場時，Y」本身已隱含每次觸發語意
- **解法**：
  - `が登場するたび` → `出場時，`（去掉「每次」）
  - `手牌に加えるたび` → `加入手牌時，`
  - MANUAL_OVERRIDES 中一律用「當…時，」，不用「每當/每次」
- **位置**：`build_data.py` sections 11、13；MANUAL_OVERRIDES 全體

---

### 問題四｜動詞語序：「自己的 + 動詞 + 受詞」

- **症狀**：「自己的公開牌庫頂部1張牌」（動詞夾在所有格和受詞中間）
- **原因**：`自分の → 自己的` 先接上，`公開し` 等動詞後替換，導致語序倒置
- **解法**：後處理 regex 把「自己的(公開|查看|棄置)(牌庫|手牌|…)」調整為「(動詞)自己的(受詞)」
- **位置**：`build_data.py` 後處理 section S

---

### 問題五｜條件句缺少「是」字

- **症狀**：「此角色攻擊角色時」（缺少「是」，不合語法）
- **應為**：「此角色是攻擊角色時」
- **原因**：`このキャラが〜キャラの場合` 中，`が` 助詞被清除，`是` 字從未被插入
- **解法**：後處理 regex 在「此角色(角色種類)角色時」前補「是」
- **位置**：`build_data.py` section S 附近

---

### 問題六｜雙角色牌括號說明文字語序混亂

- **症狀**：「此牌場地的角色的計算數量時計為1人」（語序顛倒）
- **應為**：「此牌在計算場地角色數量時計為1人」
- **原因**：括號內多個從句分別被不同 regex 翻譯，組合後語序錯亂
- **解法**：用完整 regex 匹配整個括號模式，統一改寫為固定格式
- **位置**：`build_data.py` lines ~446–449

---

### 問題七｜`に置けば使える` 翻成「時可使用」而非「後可發動」

- **症狀**：Event 牌放置條件翻成「時可使用」（語意不準確）
- **應為**：「後可發動」
- **原因**：catch-all 規則語意偏移；且含所屬前綴的 Event 牌（如「音駒のイベントカード1枚」）無法被無前綴的 regex 匹配
- **解法**：
  - `に置けば使える` → `後可發動`
  - 手牌 Event 牌放置規則允許前綴（含「音駒の」等所屬名）
- **位置**：`build_data.py` line ~532 及 section 3

---

### 問題八｜`値が(N)或以上/以下` 產生多餘 `=`

- **症狀**：「進攻値=2或以下時」（`=` 多餘）
- **原因**：section 14 先把「以下の場合」轉為「或以下時」，之後 section Q 的 `値が(N) → 値=N` 規則把剩下的「値が2」轉為「値=2」，導致「値=2或以下」
- **解法**：在 `値が(N) → 値=N` 之前，先處理 `値が(N)或以上/以下 → 値N或以上/以下`
- **位置**：`build_data.py` section Q（`或以上/以下` 規則排在精確 `=` 規則之前）

---

### 問題九｜`自分の + Area + から` → 「自己的從X區」語序錯誤

- **症狀**：「自己的從棄牌區」「自己的從接球區」
- **原因**：section O（`Area + から → 從Area`）執行後，前綴「自己的」殘留在「從」前面
- **涉及區域**：Event區、棄牌區、接球區、發球區、攻擊區、攔網區、舉球區、手牌、牌庫
- **解法**：後處理 regex：`自己的從(X區) → 從自己的(X區)`
- **位置**：`build_data.py` 後處理 section S

---

### 問題十｜`自分の手札から...をドロップすれば使える` 前綴殘留

- **症狀**：「自己的棄置手牌中的白鳥沢的牌1枚後可發動」（語序錯）
- **應為**：「從自己的手牌棄置白鳥沢的牌1張後可發動」
- **原因**：regex 只匹配 `手札から` 開頭的部分，前面的 `自分の` 被跳過，之後翻成「自己的」殘留在句首
- **解法**：新增「`自分の手札から`」前綴版本的規則，優先匹配（寫在無前綴版本之前）
- **位置**：`build_data.py` section 3（line ~210–218）

---

### 問題十一｜`相手の + Area + から` → 「對手的從X區」語序錯誤

- **症狀**：「對手的從接球區棄置…」
- **原因**：與問題九相同機制，前綴為「對手的」
- **解法**：後處理 regex 追加：`對手的從(X區) → 從對手的(X區)`
- **位置**：`build_data.py` 後處理 section S（與問題九規則相鄰）

---

### 問題十二｜`以下から1つを選んで使える` → 「選擇以下其中一項可發動的」（多餘殘留）

- **症狀**：「選擇以下其中一項可發動的」（「可發動的」是多餘殘留）
- **原因**：section 2.5 的 `で使える → 可發動的` 先把 `選んで使える` 中的 `で使える` 吃掉，導致後面的選擇規則只能匹配到 `選ん`，「可發動的」殘留句尾
- **解法**：在 `で使える → 可發動的` 之前，先完整匹配 `以下から(N)つを選んで使える → 選擇以下其中N項使用`
- **位置**：`build_data.py` section 2.5（`で使える` 規則前一行）

---

### 問題十三｜`自己的在X區中` / `對手的在X區中` 語序錯誤

- **症狀**：「自己的在攔網區中有3人時」
- **應為**：「自己的攔網區中有3人時」
- **原因**：`AREAに → 在AREA中`（section P）執行後，前綴「自己的/對手的」殘留在「在」前面
- **解法**：後處理 regex：`(自己的|對手的)在(X區)中 → \1\2中`
- **位置**：`build_data.py` 後處理 section S

---

### 問題十四｜`さらに…值に+N` → 「再+N值」語序顛倒

- **症狀**：「再+6攔網值」（數字和值名稱順序顛倒）
- **應為**：「攔網值再 +6」
- **原因**：`さらに → 再，` 先轉換，之後「値に+N」的 N 和值名稱順序錯置
- **解法**：後處理 regex：`再\+(\d+)(值名) → \2再 +\1`
- **位置**：`build_data.py` 後處理 section S

---

### 問題十五｜`ディフェンスポイント` 應翻成「防禦值」

- **症狀**：「防守點數」（語意不準確，且與其他值術語格式不一致）
- **應為**：「防禦值」（ブロックのディフェンスポイント = 攔網的防禦值）
- **解法**：`ディフェンスポイント → 防禦值`
- **位置**：`build_data.py` 專有名詞替換區

## 新增翻譯規則的 Checklist

新增規則時，請依序確認：

- [ ] 新規則放在比自己「更一般/更短」的規則**之前**
- [ ] 若有「自分の/相手の」前綴版本，先寫前綴版本再寫無前綴版本
- [ ] 若規則可能影響語序，在後處理 section S 補 fixup regex
- [ ] 執行 `python build_data.py`，確認「[OK] 生成完成，364張」
- [ ] 用 `_check.js` 或手動搜尋驗證影響到的卡片翻譯結果
- [ ] commit `cards_data.js`（不要 commit `build_data.py`）
